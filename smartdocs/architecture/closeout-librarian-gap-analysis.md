# Closeout Librarian Runtime — Gap Analysis

**Prepared during:** Closeout Librarian / Foreman-Worker hardening implementation
**Date:** 2026-06-03

This document catalogs architectural gaps discovered during the design and implementation
of the Closeout Librarian runtime flow and the Foreman/Worker behavior hardening.
Each gap is classified by priority tier.

---

## Gap Taxonomy

| Classification | Meaning |
|---|---|
| Required before implementation | Must be resolved before the feature can be used safely |
| Required during implementation | Must be implemented as part of this work |
| Future enhancement | Should be implemented, but does not block current use |

---

## Part 1 — Closeout Librarian Gaps

### G1: `librarian packet` CLI Command Missing

**Description:** The Foreman must generate a `CloseoutLibrarianPacket` via
`npm run polaris -- librarian packet <cluster-id>`. This CLI command does not yet exist.

**Why it matters:** Without this command, the Foreman must manually construct the packet
(as it did with runtime state repair in POL-283). This recreates the conditions for
Foreman improvisation, which caused the POL-288 failures.

**Recommended solution:** Implement `npm run polaris -- librarian packet <cluster-id>` in
`src/cli/`. The command should:
1. Read `current-state.json` for the active run.
2. Read `cluster-state.json` for the cluster.
3. Read all completed child compact-return results.
4. Enumerate all pending cognition notes.
5. Enumerate all affected folders and their POLARIS.md / SUMMARY.md paths.
6. Build the CloseoutLibrarianPacket and write to `.polaris/clusters/<id>/librarian-packet-<dispatch-id>.json`.
7. Print the packet path to stdout.

**Classification:** Required during implementation.

---

### G2: Finalize Has No Librarian Gate

**Description:** `npm run polaris -- finalize run` proceeds directly to PR creation
without checking whether the Closeout Librarian has completed. The Librarian step can be
bypassed by calling finalize directly.

**Why it matters:** If the Librarian is skipped (accidentally or by a non-quiet Foreman
improvising), the PR is created without documentation reconciliation. The architecture
requires that "PR creation must not occur until Librarian reconciliation succeeds."

**Recommended solution:** Add a pre-PR gate in `src/finalize/steps/08-create-pr.ts` (or
as a new step `07.5-librarian-gate.ts`) that:
1. Reads the cluster state for `librarian_dispatch_id` and `librarian_status`.
2. If missing or status is not `"success"/"partial"`, aborts with:
   `finalize aborted: Closeout Librarian has not completed for cluster <id>.`
3. Provides bypass flag `--skip-librarian-gate` (operator must explicitly opt out).

**Classification:** Required during implementation.

---

### G3: No Librarian Dispatch Boundary Enforcement

**Description:** The runtime enforces worker dispatch boundaries via `dispatch-boundary.ts`.
There is no equivalent enforcement for the Librarian dispatch.

**Why it matters:** A Foreman under token pressure might skip the Librarian step and go
directly to finalize. Without a runtime hard-fail, this is detectable only by reading
the finalize output.

**Recommended solution:** Record the Librarian dispatch ID in `current-state.json` when
the Librarian is dispatched. The finalize Librarian gate (G2) reads this field. If the
field is absent, finalize rejects unless `--skip-librarian-gate` is explicitly passed.

**Classification:** Required during implementation.

---

### G4: PR Body Does Not Include Librarian Commit

**Description:** The PR body generated by finalize does not include the Librarian commit
SHA or the documentation changes made by the Librarian.

**Why it matters:** Reviewers cannot see which POLARIS.md files were updated or which
documentation was ingested. The PR appears to contain only implementation changes.

**Recommended solution:** Update `src/finalize/steps/08-create-pr.ts` to:
1. Read the Librarian result from cluster state.
2. Include Librarian commit SHA, evidence summary, and file count in the PR body.
3. Add a "Documentation Reconciliation" section to the PR template.

**Classification:** Required during implementation.

---

### G5: No `librarian-dispatched` Telemetry Event

**Description:** The telemetry ledger has no event for when the Closeout Librarian is
dispatched. The `librarian-start` and `librarian-complete` events are defined in the
Librarian's own chain, but there is no Foreman-emitted dispatch event.

**Why it matters:** The run ledger cannot show when the Librarian was dispatched vs when
it completed. Recovery analysis (like POL-288 postmortem) cannot reconstruct Librarian
execution timeline from ledger data alone.

**Recommended solution:** Add `librarian-dispatched` and `librarian-result-validated`
events to the telemetry schema. The Foreman emits these from `polaris-run` step 08.

**Classification:** Required during implementation.

---

### G6: Librarian Cognition Notes May Be Stale

**Description:** The Librarian reads `packet.cognition_notes` which are the pending
notes at packet generation time. If a worker writes a note after the packet is generated,
the Librarian will miss it.

**Why it matters:** The Librarian is designed to run after ALL children complete. If the
packet is generated before the last worker's note is written, the Librarian has an
incomplete picture.

**Recommended solution:** The `librarian packet` CLI command (G1) should be called AFTER
`loop continue` confirms the cluster is in `cluster-complete` state — not before. The
packet generation should scan `pending/` at that point to capture all notes.

**Classification:** Required before implementation.

---

### G7: No Librarian Re-dispatch Protocol

**Description:** If the Librarian fails (status "failure") or times out, the Foreman
must re-dispatch it. But there is no defined re-dispatch sequence, and the existing
packet may be stale.

**Why it matters:** A failed Librarian with a stale packet re-dispatched as-is may
produce incorrect results. The packet must be regenerated.

**Recommended solution:** When re-dispatching the Librarian:
1. Generate a new packet (new dispatch_id, current state).
2. Use the same cluster-id.
3. The per-cluster-invariant holds: only one active Librarian per cluster.
4. Record the re-dispatch in telemetry as `librarian-redispatched` with prior dispatch_id.

**Classification:** Required during implementation.

## Part 2 — Foreman/Worker Hardening Gaps

### G8: `stdio: "inherit"` in terminal-cli Adapter

**Description:** `src/loop/adapters/terminal-cli.ts` uses `stdio: "inherit"` when
spawning worker processes, which merges worker stdout/stderr into the Foreman's process.
This caused the 21.7M cached token burn in POL-283.

**Why it matters:** This is the architectural root cause of the Foreman's context
explosion. Role file instructions alone cannot fix this — the adapter must be changed.

**Recommended solution:** Replace `stdio: "inherit"` with CompactReturn-only extraction:
1. Use `stdio: "pipe"` (or `"pipe"/"inherit"/"inherit"` for stderr visibility).
2. Buffer the worker's stdout.
3. Extract only the last JSON line matching the CompactReturn schema.
4. Return the CompactReturn to the Foreman; discard everything else.

This eliminates the possibility of worker output contaminating the Foreman context.

**Classification:** Required during implementation.

---

### G9: No Worker Scope Fidelity Check

**Description:** When `loop continue` runs after a worker completes, it does not verify
that the worker's commit only contains files within `allowed_scope`.

**Why it matters:** The POL-288 worker (Copilot/POL-287) staged `current-state.json` in
its delivery commit despite explicit instructions not to. The Foreman only discovered this
by reading the worker transcript — a practice that violates the CHECKPOINT gate.

**Recommended solution:** In `src/loop/continue.ts`, after receiving the CompactReturn:
1. Run `git show --name-only <commit_sha>` to list files in the commit.
2. Compare against `packet.allowed_scope` and `packet.prohibited_write_paths`.
3. If violations found: emit `worker-scope-fidelity` telemetry event, escalate to operator
   (do not advance queue automatically).

**Classification:** Required during implementation.

---

### G10: No `worker complete` CLI Command

**Description:** Workers write their CompactReturn by writing directly to
`packet.expected_result_path`. There is no CLI command that validates the result before
accepting it.

**Why it matters:** Direct file writes bypass validation. A malformed CompactReturn or a
result with incorrect fields can cause `loop continue` to fail in unpredictable ways.

**Recommended solution:** Add `npm run polaris -- worker complete <result-file>` to
`src/cli/worker.ts`. This command:
1. Validates the result against the CompactReturn schema.
2. Verifies the dispatch_id and child_id match the active dispatch.
3. Writes the result to the expected path (not the worker).
4. Appends telemetry event.

Workers call this command; direct result file writes are logged as warnings.

**Classification:** Required during implementation.

---

### G11: Foreman Live Repair Has No Detection Mechanism

**Description:** The Foreman can patch state files, create cluster-state.json, and amend
commits without any telemetry event capturing this activity. Recovery cycles are invisible
to the ledger.

**Why it matters:** The POL-288 postmortem required reading the full session transcript to
reconstruct 4 recovery cycles. The ledger had no `recovery-cycle` events. Future operators
cannot evaluate run health without reading transcripts.

**Recommended solution:**
1. Add `recovery-action` telemetry event with fields: `type` (state-patch, commit-amend,
   etc.), `affected_file`, `run_id`, `timestamp`.
2. CLI commands that perform recovery (future `polaris state repair` command) emit this
   automatically.
3. The ledger aggregates `recovery-count` per run_id.

**Classification:** Future enhancement (requires CLI repair commands first).

---

### G12: Worker Can Call `loop continue` Directly

**Description:** Workers are prohibited from calling `polaris loop continue`, but there
is no runtime enforcement. A worker that calls `continue` would advance the dispatch
boundary without the Foreman's involvement.

**Why it matters:** Worker-initiated `loop continue` would corrupt the epoch counters and
potentially allow a worker to mark a different child as complete.

**Recommended solution:** The dispatch-boundary check in `dispatch-boundary.ts` should
validate that `continue` is called from the same session (run context) as the original
dispatch. Workers run in separate sessions; cross-session `continue` calls should hard-fail.

**Classification:** Future enhancement (requires session identity tracking).

---

### G13: Missing Heartbeat Liveness Supervision in Chain

**Description:** The polaris-run chain does not include an explicit heartbeat supervision
step. The Foreman monitors liveness by watching for CompactReturn, but there is no defined
procedure for what to do when a heartbeat expires during waiting.

**Why it matters:** Without a defined procedure, the Foreman improvises — which in POL-283
led to the Foreman monitoring via `ps` and reading worker output directly.

**Recommended solution:**
1. Add a `heartbeat-supervisor` check to the polaris-run chain (as a parallel monitor, not
   a blocking step).
2. The supervisor emits `heartbeat-stale` event when last_heartbeat > 120s.
3. The Foreman responds with the defined escalation format (not improvisation).

**Classification:** Future enhancement.

### G16: Assumption: Librarian Runs in Same Session as Foreman

**Description:** The current design assumes the Librarian is dispatched as a subagent
within the Foreman's session. If the Foreman session ends before the Librarian completes,
there is no continuation protocol.

**Why it matters:** Context budget exhaustion can end a Foreman session mid-run.
If this happens after cluster-complete but before Librarian completes, the next session
would need to detect the orphaned Librarian and handle it.

**Recommended solution:** Add `librarian_status` field to `current-state.json` as a
transient telemetry hint. When a new session resumes a run, it first validates canonical
cluster and run state via the run management API or canonical store. Only if canonical
state grants finalize authority should the session consult `librarian_status` as a
compatibility hint for re-dispatch decisions. `current-state.json` must not be treated
as the authoritative source for whether the Librarian has completed — only the canonical
run state and the actual sealed result file at `result_path` carry that authority.

**Classification:** Required during implementation.

---

### G17: No Maximum Librarian Dispatch Count per Cluster

**Description:** There is no defined limit on how many times the Librarian can be
re-dispatched per cluster. An operator could re-dispatch indefinitely.

**Why it matters:** Unlike workers (which have a replacement ladder), the Librarian has
no defined escalation path after repeated failures.

**Recommended solution:** After 2 failed Librarian dispatches, the Foreman should present
a final option to skip with operator acceptance (explicitly recording the degraded state).
The cluster should not be indefinitely blocked by a failing Librarian.

**Classification:** Future enhancement.

## Critical Path

The following gaps must be addressed before the Closeout Librarian can run safely:

**G6** (cognition notes may be stale at packet generation) — must be resolved in packet
generation timing. The `librarian packet` command must run after `cluster-complete` is
confirmed.

Then during implementation:

1. **G1** — Implement `npm run polaris -- librarian packet <cluster-id>` CLI command
2. **G2 + G3** — Add Librarian gate to finalize, record librarian dispatch in state
3. **G8** — Fix terminal-cli adapter output filtering
4. **G9** — Add worker scope fidelity check to `continue.ts`
5. **G7 + G16** — Define re-dispatch protocol and handle orphaned Librarian in session resume
6. **G4 + G5** — Add Librarian metadata to PR body and telemetry
7. **G14 + G15** — POLARIS.md and provenance files for new skill directory
